# Development Rules

프로젝트 전체에서 반드시 지켜야 하는 개발 규칙

---

# Rule 1

**기능 하나 = Commit 하나**

---

# Rule 2

Commit 후 반드시 GitHub에 Push한다.

---

# Rule 3

기획 없이 코드를 작성하지 않는다.

---

# Rule 4

데이터 구조를 먼저 설계한 후 구현한다.

---

# Rule 5

`bot.py`는 Discord 연결만 담당한다.

게임 로직은 별도 모듈에서 관리한다.

---

# Rule 6

모든 시스템은 확장 가능하도록 설계한다.

임시 코드가 프로젝트에 남지 않도록 한다.

---

# Rule 7

Discord Token, API Key 등 모든 Secret은 GitHub에 업로드하지 않는다.

`.env` 파일을 사용하여 관리한다.

---

# Rule 8

새로운 기능을 구현하기 전에 관련 문서를 먼저 수정한다.

---

# Rule 9

버전 규칙

* v0.x : 프로토타입
* v1.x : 정식 서비스

---

# Rule 10

새로운 아이디어는 바로 구현하지 않는다.

먼저 `10_IDEAS.md`에 기록한 뒤, 검토 후 개발한다.

---

# 프로젝트 원칙

**"빠르게 만드는 것보다 오래 유지할 수 있는 구조를 만든다."**

## Rule 11

메키는 Data Driven Development를 지향한다.

게임 데이터(몬스터, 아이템, 스킬, 퀘스트 등)는
가능한 한 코드가 아니라 데이터 파일(JSON)로 관리한다.

새로운 콘텐츠를 추가할 때는
코드를 수정하지 않고 데이터만 추가할 수 있도록 설계한다.

---

# Rule 12

게임 데이터 및 세계관은 프로젝트의 핵심 자산이다.

사용자가 작성한 설정(Description, Flavor Text, 스토리, 세계관 등)은
임의로 수정하거나 축약하지 않는다.

문장 개선이나 수정이 필요한 경우에는
반드시 기존 내용을 유지한 상태에서 수정안을 제안하고,
사용자의 승인 후 변경한다.

---

# Rule 13

모든 기능 구현은 다음 순서로 진행한다.

1. 구현 계획 설명
2. 파일 생성
3. 코드 작성
4. 실행 및 테스트
5. Commit
6. Push

각 단계가 완료되었음을 확인한 후
다음 단계를 진행한다.

한 번에 여러 단계를 진행하지 않는다.

---

# Rule 14

개발자는 프로젝트 외의 작업을 병행할 수 있음을 고려한다.

새로운 파일 생성, 폴더 생성, 실행 명령 등
중간 과정이 필요한 작업은
체크리스트 형태로 안내한다.

예시

[ ] 파일 생성
[ ] 코드 작성
[ ] 실행
[ ] 결과 확인
[ ] Commit
[ ] Push

---

# Rule 15

반복적으로 사용하는 검증은
Python REPL보다 재사용 가능한 스크립트를 우선 사용한다.

확인용 코드는 tools 폴더에 작성하여
동일한 검증을 언제든지 한 번의 실행으로 수행할 수 있도록 한다.

---

# Rule 16

새로운 기능을 구현할 때는
먼저 전체 구조와 목적을 설명한 후 구현을 시작한다.

사용자가 현재 구현하는 기능의 역할과
프로젝트 전체에서 어떤 위치를 차지하는지 이해할 수 있도록 설명한다.

단순히 코드를 제공하는 것이 아니라
초보 개발자가 프로젝트 구조를 이해하며 개발할 수 있도록 안내한다.

# Rule 17

하나의 Commit은 하나의 기능만 포함한다.

기능 구현 중 개선 아이디어가 떠오르더라도
현재 Commit에 포함하지 않는다.

현재 기능을 완료한 후
다음 Commit에서 구현한다.
